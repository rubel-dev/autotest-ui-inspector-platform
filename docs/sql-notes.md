# SQL Notes

## Purpose

Phase 2 defines the PostgreSQL foundation for AutoTest UI Inspector Platform. The application will access this database through raw, parameterized SQL repositories only. No ORM is planned or allowed.

## Files

- `backend/sql/schema.sql` creates extensions, tables, relationships, and constraints.
- `backend/sql/indexes.sql` adds indexes for authentication, ownership scoping, filtering, sorting, logs, and analytics.
- `backend/sql/seed.sql` inserts realistic demo data for dashboard development.

## Major Schema Decisions

### UUID Primary Keys

Every table uses UUID primary keys generated by PostgreSQL through `gen_random_uuid()` from `pgcrypto`. This keeps IDs opaque in public APIs and avoids exposing sequential business volume.

### Owner-Scoped Data

Projects belong to users through `projects.owner_id`. Most user-facing data is reached through projects, so future repository queries must always scope reads and writes by the authenticated user.

Example ownership pattern:

```sql
SELECT tc.*
FROM test_cases tc
JOIN projects p ON p.id = tc.project_id
WHERE tc.id = $1
  AND p.owner_id = $2;
```

### Cascading Project Data

Project-owned records use `ON DELETE CASCADE` so deleting a project removes pages, elements, test cases, runs, logs, and activity tied to the project. This makes cleanup predictable for a portfolio SaaS project.

### Duplicated `project_id` on `ui_elements` and `test_runs`

`ui_elements` could be reached through `project_pages`, and `test_runs` could be reached through `test_cases`, but both tables also store `project_id`. This is intentional:

- Faster project-scoped filters.
- Simpler analytics joins.
- Easier authorization queries.
- Cleaner indexes for dashboard tables.

Composite foreign keys ensure the duplicated `project_id` matches the related page or test case.

### JSONB Metadata

`ui_elements`, `run_logs`, and `activity_logs` include `metadata jsonb` for flexible details that do not deserve first-class columns yet. This allows the inspector and execution simulator to store useful structured context without changing the schema for every minor attribute.

### Status and Type Checks

Important enums are enforced with `CHECK` constraints:

- User roles.
- UI element types.
- Test case priority and status.
- Test step actions.
- Test run statuses.
- Log severity and status.

The backend should mirror these values in Pydantic schemas.

### Lowercase Emails

The `users` table requires lowercase email values. The backend should normalize emails before insert and login lookup.

## Index Strategy

Indexes support the first real API screens:

- Login lookup by email.
- Project list by owner and created or updated time.
- Project search by lowercase name.
- Page list by project and inspection time.
- Element filters by project and type.
- Test case filters by project, status, and priority.
- Run history by project, test case, status, and date.
- Log lookup by run, severity, status, and message.
- Activity feed by user or project.

Search is currently supported with simple `lower(column) LIKE` patterns. If search becomes a larger feature, enable `pg_trgm` and add trigram GIN indexes for names, URLs, selectors, and log messages.

## Important Queries Needed Later

### Project List With Aggregated Counts

Needed by `/api/v1/projects`.

This query should return each project with:

- Page count.
- UI element count.
- Test case count.
- Test run count.
- Last run status.

Expected shape:

```sql
WITH project_stats AS (
    SELECT
        p.id AS project_id,
        count(DISTINCT pp.id) AS page_count,
        count(DISTINCT ue.id) AS element_count,
        count(DISTINCT tc.id) AS test_case_count,
        count(DISTINCT tr.id) AS run_count
    FROM projects p
    LEFT JOIN project_pages pp ON pp.project_id = p.id
    LEFT JOIN ui_elements ue ON ue.project_id = p.id
    LEFT JOIN test_cases tc ON tc.project_id = p.id
    LEFT JOIN test_runs tr ON tr.project_id = p.id
    WHERE p.owner_id = $1
    GROUP BY p.id
)
SELECT p.*, ps.page_count, ps.element_count, ps.test_case_count, ps.run_count
FROM projects p
JOIN project_stats ps ON ps.project_id = p.id
WHERE p.owner_id = $1
ORDER BY p.created_at DESC
LIMIT $2 OFFSET $3;
```

During implementation, this may be optimized with separate aggregate CTEs to avoid row multiplication.

### Test Case Detail With Steps

Needed by `/api/v1/test-cases/{id}`.

Use owner scoping through `projects`, then fetch ordered steps:

```sql
SELECT ts.*
FROM test_steps ts
JOIN test_cases tc ON tc.id = ts.test_case_id
JOIN projects p ON p.id = tc.project_id
WHERE tc.id = $1
  AND p.owner_id = $2
ORDER BY ts.step_order ASC;
```

### Logs Search

Needed by `/api/v1/logs`.

Logs should include project and test case context:

```sql
SELECT
    rl.*,
    tr.status AS run_status,
    tc.name AS test_case_name,
    p.name AS project_name
FROM run_logs rl
JOIN test_runs tr ON tr.id = rl.run_id
JOIN test_cases tc ON tc.id = tr.test_case_id
JOIN projects p ON p.id = tr.project_id
WHERE p.owner_id = $1
  AND ($2::text IS NULL OR lower(rl.message) LIKE '%' || lower($2) || '%')
ORDER BY rl.created_at DESC
LIMIT $3 OFFSET $4;
```

### Analytics Overview With CTEs

Needed by `/api/v1/analytics/overview`.

The backend should use a CTE-based query that scopes projects first, then runs and failures:

```sql
WITH owned_projects AS (
    SELECT id
    FROM projects
    WHERE owner_id = $1
),
owned_runs AS (
    SELECT tr.*
    FROM test_runs tr
    JOIN owned_projects op ON op.id = tr.project_id
),
totals AS (
    SELECT
        (SELECT count(*) FROM owned_projects) AS project_count,
        (SELECT count(*) FROM test_cases tc JOIN owned_projects op ON op.id = tc.project_id) AS test_case_count,
        count(*) AS run_count,
        round(
            100.0 * count(*) FILTER (WHERE status = 'passed') / nullif(count(*), 0),
            2
        ) AS pass_rate
    FROM owned_runs
)
SELECT *
FROM totals;
```

### Common Failure Reasons

Needed by analytics and logs:

```sql
SELECT failure_reason, count(*) AS failure_count
FROM test_runs tr
JOIN projects p ON p.id = tr.project_id
WHERE p.owner_id = $1
  AND tr.status = 'failed'
  AND failure_reason IS NOT NULL
GROUP BY failure_reason
ORDER BY failure_count DESC, failure_reason ASC
LIMIT $2;
```

## Repository Implementation Rules

- Keep SQL inside repository modules only.
- Use placeholders such as `$1`, `$2`, `$3` with asyncpg.
- Build dynamic filters with a parameter list and controlled SQL fragments.
- Allowlist sort columns before placing them in SQL.
- Use transactions for multi-write operations:
  - Register user and activity log.
  - Create test case and steps.
  - Inspect page and replace or insert UI elements.
  - Create test run and run logs.
- Return predictable row shapes for service-level serialization.

## Seed Data Notes

The seed data uses fixed UUIDs so tests and documentation can reference stable records. Password hashes are placeholders and should be replaced with real bcrypt hashes during backend implementation or test fixture setup.
